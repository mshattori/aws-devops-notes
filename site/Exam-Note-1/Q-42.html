問題42 正解

シナリオ（日本語要約）

  • あるビジネスは Go 言語ベースのオンプレミスアプリケーション を AWS へマイグレーションしようとしている。
  • 企業は移行を機に、AWS が提供する新しい機能を積極的に活用したいと考えている。
  • 開発チームは、現在の本番環境（Blue）と新環境（Green）の 2 つの同一環境を並行稼働 させ、A/B テストで新機能や変更を検証できるようにしたい。

質問

  • 上記要件を満たすために、DevOps エンジニアが実装すべき 最も適切なソリューション はどれか。

選択肢（日本語訳）

  • 選択肢1

    • アプリケーションを Amazon Lightsail 上にホストし、アプリケーションを ZIP 形式で Amazon S3 バケット にアップロードする。
    • バケットを使ってアプリケーションの新バージョンを実装し、Lightsail のデプロイメントオプションで管理する。

  • 選択肢2

    • アプリケーションを Amazon EC2 インスタンス 上にホストし、その AMI を作成する。
    • その AMI をもとに Launch Template と Auto Scaling グループを構成し、Application Load Balancer をプロビジョニングしてトラフィックを分散する。
    • アプリケーションの変更時には新しい AMI を作成し、EC2 インスタンスリフレッシュをトリガーする。

  • 選択肢3（正解）

    • アプリケーションを AWS Elastic Beanstalk 上にホストする。
    • アプリケーションを ZIP 形式で Amazon S3 バケット にアップロードし、そのバケットを利用して新バージョンをデプロイする。
    • Elastic Beanstalk の デプロイメントオプション（Blue/Green / Rolling / Immutable など） を利用してデプロイを管理する。

  • 選択肢4

    • アプリケーションコードを AWS CodeArtifact でホストし、AWS CodeDeploy を使って EC2 フリートにデプロイする。
    • トラフィック分散には Application Load Balancer を用い、アプリケーション変更時は CodeArtifact に新バージョンをアップロードして新しい CodeDeploy デプロイメントを作成する。

正解と注釈（選択肢3）

  • キーワード

    • AWS Elastic Beanstalk
    • Blue/Green Deployment
    • S3 によるアプリケーションバージョン管理

  • この解決策が要件を満たす理由

    • Elastic Beanstalk は Blue/Green デプロイ やローリングデプロイオプションを標準で提供しており、2 つの同一環境（本番とテスト）を並行稼働 させる構成が簡単に実現できる。
    • アプリケーションパッケージを S3 に保存しておくことで、バージョンロールバックや A/B テスト時の環境切り替えが容易になる。
    • 基盤となる EC2 / Auto Scaling / Load Balancer の管理を Beanstalk が抽象化してくれるため、DevOps チームはアプリケーションに集中できる。

  • 他の選択肢が不適切な理由（要点）

    • 選択肢1（Lightsail）
      • Lightsail は簡易ホスティングに適したサービスであり、本格的な Blue/Green デプロイや A/B テスト機能 は提供していない。
    • 選択肢2（EC2 + Auto Scaling 手動構成）
      • Blue/Green やロールバックの仕組みをすべて自前で実装する必要があり、運用負荷が高い。
    • 選択肢4（CodeArtifact + CodeDeploy）
      • コンポーネントが多く、マイグレーション初期段階としてはやや複雑。Blue/Green 自体は実装可能だが、Elastic Beanstalk を使う場合に比べて セットアップと運用のオーバーヘッド が大きい。
