問題44 正解

シナリオ（日本語要約）

  • スマートホームデバイスを開発する会社が、Web / Mobile / IoT / サードパーティ API などを処理する 複数の AWS Lambda 関数 からなるサーバーレス API 群を構築している。
  • 現在の CI/CD パイプラインは AWS CodePipeline + AWS CodeBuild を利用しており、各 Lambda 関数を 順番にビルド・テスト・パッケージング・デプロイ している。
  • コード変更が発生してからパイプラインが完了するまでの時間が長く、デプロイプロセスの高速化が求められている。
  • パイプライン実行のトリガーには Amazon CloudWatch Events (EventBridge) を使用し、SysOps チームがパイプラインおよびリソースのパフォーマンスを監視している。

質問

  • 独立した複数 Lambda 関数のデプロイをスピードアップするために、DevOps エンジニアは どのような CodePipeline 構成 を取るべきか。

選択肢（日本語訳）

  • 選択肢1

    • CodeBuild パイプラインのビルド環境を、メモリ・vCPU・ディスク容量が多い より大きなコンピュートタイプ へアップグレードする。

  • 選択肢2

    • 複数のビルドを並列実行できるように、マルチプロセッシングオプション付きのカスタム CodeBuild 実行環境 を用意する。

  • 選択肢3（正解）

    • 各 Lambda 関数のアクションに対して、CodePipeline で 同じ runOrder 値 を指定する。
    • これにより、複数の Lambda 関数に対するビルド／デプロイアクションが 並列に実行 される。

  • 選択肢4

    • Docker レイヤーキャッシュモードを使用して CodeBuild 内のローカルキャッシュを有効にし、ビルド時間を短縮する。

正解と注釈（選択肢3）

  • キーワード

    • CodePipeline runOrder
    • 並列アクション実行
    • 独立した Lambda 関数の同時デプロイ

  • この解決策が要件を満たす理由

    • CodePipeline のアクションに同じ runOrder を設定すると、同一ステージ内でそれらのアクションが並列実行 される。
    • 問題文の Lambda 関数は「独立した」関数であるため、並列実行しても依存関係の問題がなく、パイプライン全体の所要時間を直接短縮 できる。
    • 既存のインフラを変えずに、パイプライン定義の変更だけで高速化できる点も、最も適切な解として評価される。

  • 他の選択肢が不適切な理由（要点）

    • 選択肢1・2・4 はビルド環境をチューニングする案だが、問題の本質は「独立した関数を順番に処理していること」にある。
    • いずれもアクションを直列から並列へ変えるわけではないため、構造的なボトルネック解消につながらない。
