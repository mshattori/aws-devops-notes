Q-10

問題10 不正解

シナリオ（日本語要約）

  • ある企業では、DevOps エンジニアが一部の Amazon EC2 Windows インスタンス に接続できなくなる事象や、ブート問題が発生している。
  • これまでは、再接続できるようにするまでに多くの 手作業のトラブルシューティング手順 が必要だった。
  • チームは、会社の VPC 内の障害 EC2 インスタンスを自動復旧 するソリューションを設計する任務を与えられた。
  • コンプライアンス要件として、以下を満たす必要がある。
    • ネットワーク設定ミス、RDP 問題、ファイアウォール設定などが原因で 到達不能になった EC2 インスタンスを自動的に修復 すること。
    • aws.trustedadvisor を使ってベストプラクティスを継続的に監視すること。

質問

  • 上記要件を満たすために、DevOps エンジニアが実装すべき 最も適切なソリューション はどれか。

選択肢（日本語訳）

  • 選択肢1

    • AWS Config と AWS Systems Manager Session Manager を組み合わせて、EC2 Linux / Windows インスタンス上の問題を自己診断・トラブルシュートする。
    • Amazon CloudWatch、AWS Lambda、Systems Manager Run Command を用いた監視システムを構築し、障害 EC2 インスタンスを自動的に監視・復旧する。

  • 選択肢2（自分が選んだ不正解）

    • AWS Trusted Advisor を使用して、EC2 インスタンスに関する一般的なセキュリティおよびパフォーマンスの問題を監視・解決し、コンプライアンスと自動復旧を実現する。

  • 選択肢3（正解）

    • EC2 Linux / Windows Server インスタンス上の問題を診断・トラブルシュートするために EC2Rescue ツール を利用する。
    • AWS Systems Manager Automation と AWSSupport-ExecuteEC2Rescue ドキュメント を使用して、このツールの実行を自動化し、到達不能なインスタンスの復旧プロセスを自動化する。

  • 選択肢4

    • AWS Config と AWS Systems Manager State Manager を統合し、EC2 Linux / Windows インスタンス上の問題を自己診断・トラブルシュートする。
    • Systems Manager Maintenance Windows を使用して復旧プロセスを自動化する。

正解の選択肢と注釈（選択肢3）

  • キーワード

    • EC2Rescue / AWSSupport-ExecuteEC2Rescue
    • AWS Systems Manager Automation
    • インスタンス到達不能時の自動復旧

  • この解決策が要件を満たす理由

    • EC2Rescue は、ネットワーク設定ミスや RDP / ファイアウォール問題などにより到達不能になった EC2 インスタンスを診断・修復するための 専用ツール であり、本問で示された障害内容に最も適している。
    • Systems Manager Automation と AWSSupport-ExecuteEC2Rescue を組み合わせることで、CloudWatch や EventBridge からトリガーされた際に 標準化された復旧ランブック を自動実行でき、手作業の復旧ステップを削減できる。
    • 併せて Trusted Advisor を利用すれば、ベストプラクティス違反（セキュリティ / コスト / パフォーマンスなど）を継続的にチェックしつつ、実際の障害発生時には EC2Rescue による自動復旧を行う構成にできる。

  • 他の選択肢が不適切な理由（要点）

    • 選択肢1
      • Config + Session Manager + Run Command は汎用的な構成だが、到達不能なインスタンスのブート問題や RDP 問題を自動修復する専用ランブック に触れていない。
      • どのような診断・修復処理を実行するかが不明確であり、ベストプラクティスとして推奨される EC2Rescue ベースのアプローチより抽象度が高い。
    • 選択肢2（自分が選んだ不正解）
      • Trusted Advisor はあくまでアカウントレベルのベストプラクティスチェック（セキュリティ、コスト最適化、パフォーマンスなど）を行うサービスであり、個々のインスタンスのブート問題や RDP 設定を自動的に修復する機能は持たない。
      • 復旧の具体的な自動化（ランブック、Automation ドキュメント）が示されておらず、「自動復旧を提供する」と記載されている点が誤り。
    • 選択肢4
      • State Manager と Maintenance Windows は構成ドリフト修正や定期パッチ適用には適しているが、到達不能なインスタンスのオンデマンド復旧 には直接結びつきにくい。
      • 本問で求められている 障害発生時の即時トラブルシュート + 修復 という要件に対しては、EC2Rescue + Automation がより直接的な解決策となる。

不正解と表示されている場合の追加注釈

  • 自分が選んだ不正解の選択肢

    • 選択肢2：AWS Trusted Advisor だけで監視と自動復旧を行う構成 を選択した。

  • この選択肢が要件を満たさない理由

    • Trusted Advisor は チェックとレポート（推奨事項の提示） が主目的であり、EC2 インスタンスに対して直接的な 修復アクションを自動実行する仕組みではない。
    • ネットワーク設定ミスや RDP 問題などの具体的な OS / インスタンスレベルの障害は、Trusted Advisor ではなく EC2Rescue や Systems Manager Automation で対応すべき領域。
    • コンプライアンス要件として求められている「到達不能になったインスタンスを自動的に修復する」点を満たしておらず、誤った期待を置いている。

  • 学習ポイント

    • Trusted Advisor = ベストプラクティスチェック、EC2Rescue + SSM Automation = 障害インスタンスの自動修復 という役割分担を明確に覚える。
    • EC2 のブート失敗や RDP 接続不可といった問題が出てきたら、まず AWSSupport-ExecuteEC2Rescue Automation ドキュメント を思い出す。
    • 「手作業の復旧手順を自動化したい」というキーワードがある場合、Systems Manager Automation / Runbook を使った設計を優先的に検討する。
