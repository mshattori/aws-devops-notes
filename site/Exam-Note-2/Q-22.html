Q-22

問題22 正解

シナリオ（日本語要約）

  • ある IT コンサル企業は、複数の AWS アカウントを AWS Organizations で管理しており、組織単位 (OU) ごとに部署やチームを分けている。
  • ある日、IT セキュリティチームが、事前承認なしに第三者の AWS アカウントが突然組織に追加されていた疑い を報告した。
    この外部アカウントには高い権限が付与されていたが、幸い実害は発生していない。
  • 会社は、今後 AWS アカウントや Organizations 設定に対する変更が行われた場合には、即座に通知を受け取れる監視体制 を構築したい。
  • 要件：
    • AWS Organizations 関連の API コールやコンソール操作を詳細に記録する。
    • 組織構成・ポリシーに対する変更を継続的に監視し、SNS / EventBridge によるアラート を受け取る。
    • Select TWO。

質問

  • 上記要件を満たすための 最も適切な監視構成 の組み合わせはどれか。

選択肢（日本語訳）

  • 選択肢1（正解）

    • 新しい Amazon CloudTrail トレイル を作成し、Organizations へのすべての API コール（Organizations コンソールからの操作を含む）を記録する。
    • さらに、AWS Organizations API へのコードベースの呼び出しもすべて追跡する。
    • Amazon EventBridge と Amazon SNS を統合 し、組織内で管理者が指定したアクションが発生した際にイベントを生成し、SNS を通じて通知を送信する。

  • 選択肢2

    • AWS Marketplace からサードパーティの監視ツールを導入し、侵害が検出された場合にアラートを送信させる。
    • 侵害の詳細は AWS GuardDuty で分析し、SNS で管理者へ通知する。

  • 選択肢3

    • AWS Systems Manager と Amazon EventBridge を使用して組織全体の変更を監視し、アカウントの新しいアクティビティや設定変更を通知する。

  • 選択肢4

    • CloudWatch Dashboard を作成して組織への変更を監視し、新しい SNS トピックを作成して通知を送る。

  • 選択肢5（正解）

    • AWS Config を利用して AWS Organizations のコンプライアンスを監視する。
    • SNS トピックまたは EventBridge ルールを設定し、Organizations の設定やアカウント構成に変更があった場合にアラートを送信する。

正解の選択肢と注釈（選択肢1・5）

  • キーワード

    • CloudTrail で Organizations API を完全記録
    • EventBridge + SNS によるイベント通知
    • AWS Config による Organizations 構成変更の監視

  • この組み合わせが要件を満たす理由

    • CloudTrail は、Organizations API を含むすべての管理プレーン API コールを記録できるため、いつ・誰が・どのようにアカウントを追加 / 削除 / 変更したか を完全にトレースできる。
    • CloudTrail を EventBridge と統合すれば、特定の Organizations イベント（例：CreateAccount, InviteAccountToOrganization, AttachPolicy など）発生時に リアルタイムで SNS 通知や自動ワークフロー を起動できる。
    • AWS Config は Organizations リソースの構成状態を継続的に評価し、想定外のアカウント追加やポリシー変更 を検出して SNS / EventBridge でアラートを送れる。
    • CloudTrail（操作ログ）と Config（現在の構成・コンプライアンス）の組み合わせにより、Who / When / What をフルカバーする監査 + 監視体制 を実現できる。

  • 他の選択肢が不適切な理由（要点）

    • 選択肢2（サードパーティツール + GuardDuty）
      • GuardDuty は脅威検出サービスであり、Organizations の構成変更やアカウント追加そのものを直接監視するわけではない。
      • サードパーティツール導入はコスト・運用面でオーバーヘッドが大きく、「MOST appropriate」な選択とは言い難い。
    • 選択肢3（Systems Manager + EventBridge）
      • Systems Manager はパッチ適用・オートメーションなど運用タスク向けであり、Organizations 管理の監査・記録には適さない。
    • 選択肢4（CloudWatch Dashboard + SNS）
      • Dashboard はメトリクス可視化用であり、Organizations 構成変更の検出には向かない。
      • 根本的なイベントソースとして CloudTrail / Config を利用していないため、要件を満たさない。

学習ポイント

  • AWS アカウント / Organizations の変更監視では、
    • CloudTrail → 管理プレーン API コールの記録と EventBridge 連携
    • AWS Config → リソース構成・ポリシーのコンプライアンス評価と SNS / EventBridge 通知
      の組み合わせが基本パターン。
  • 「第三者アカウントが勝手に追加された疑い」などのシナリオでは、誰が招待 / 作成したかを追跡する CloudTrail と、現在の OU / SCP / アカウント構成を監視する Config の両方を使う構成を思い出す。
