問題33 正解

シナリオ（日本語要約）

  • 大手ソフトウェア開発企業が、クラウドベースアプリケーションの 1 つで GitHub をソースコードリポジトリ として使用している。
  • 新しい要件として、開発 / テスト / 本番環境に対する継続的デプロイメントパイプライン（CI/CD）の自動化 が求められている。
  • 追加要件：
    • 各コード変更に対して セキュリティコードレビュー を行い、PII や機密データの漏洩がないことを確認 する。
    • すべての変更は ユニットテスト + 機能テスト を通過する必要がある。
    • GitHub へのプッシュで自動的に CI/CD パイプラインが起動し、devops-administrators@tutorialsdojo.com へメール通知を送る。
    • ビルド / デプロイ失敗時にも通知を送付する。
    • テスト後には Amazon S3 へのステージング/本番デプロイに対する手動承認ステップ を含める。
    • ソリューションは CI/CD のベストプラクティス に忠実であること。

質問

  • 上記要件すべてを満たすために、DevOps エンジニアが実装すべきオプションはどれか。

選択肢（日本語訳）

  • 選択肢1（正解）

    • AWS CodePipeline に、特定ブランチへの新しいコードコミットをトリガーとするパイプラインを構成する。
    • パイプライン内に、セキュリティレビュー、ユニットテスト、機能テスト、手動承認 の各ステージを設定する。
    • Amazon EventBridge を用いてパイプラインステージの状態変化イベントを検出し、devops-administrators@tutorialsdojo.com へメールを送信する SNS 通知を構成する。

  • 選択肢2

    • master ブランチへのコミットをトリガーとする CodePipeline を構成し、同様のステージ（セキュリティレビュー、ユニットテスト、機能テスト、手動承認）を追加する。
    • パイプラインステージの状態変化を CloudWatch Logs 経由で検知 し、Amazon SES を使用してメール通知を送信する。

  • 選択肢3

    • 特定ブランチへのコミットをトリガーとする CodePipeline を構成し、必要なステージを追加する。
    • パイプラインの状態変化を EventBridge で検出し、Amazon SES を通じてメールを送信する。

  • 選択肢4

    • master ブランチへのコミットをトリガーとする CodePipeline を構成し、必要なステージを追加する。
    • パイプラインイベントを CloudTrail で検出し、SNS を通じて通知する。

正解の選択肢と注釈（選択肢1）

  • キーワード

    • GitHub Webhook トリガー付き AWS CodePipeline
    • セキュリティレビュー / テスト / 手動承認ステージ
    • EventBridge + SNS によるパイプラインイベント通知

  • この解決策が要件を満たす理由

    • CodePipeline は GitHub とネイティブ連携でき、特定ブランチへのコミットをトリガーにして CI/CD パイプラインを自動起動 できる。
    • ステージとして、静的解析ツールやコードスキャンを利用した セキュリティレビュー、ユニットテスト / 機能テスト、本番デプロイ前の 手動承認アクション をシンプルに組み込める。
    • CodePipeline のステートチェンジイベントは EventBridge に自動的に発行されるため、EventBridge ルールで特定ステージの成功 / 失敗をフィルタし、ターゲットの SNS トピック経由で メール通知を自動送信 できる。
    • これにより、GitHub → CodePipeline → テスト → 手動承認 → S3 デプロイ → 通知 までが、一貫した ベストプラクティスに沿った CI/CD フロー として実現される。

  • 他の選択肢が不適切な理由（要点）

    • 選択肢2（CloudWatch Logs + SES）
      • パイプラインイベントを CloudWatch Logs から検出するのは間接的であり、EventBridge という専用イベントバスを利用する方がシンプルかつリアルタイム。
      • SES での送信管理も必要となり、SNS に比べ設定が煩雑。
    • 選択肢3（SES 通知）
      • EventBridge 連携自体は良いが、SES を直接ターゲットにするよりも、SNS をメールチャネル + 将来の別チャネル通知のハブとして使う 方が拡張性が高く、ベストプラクティスに近い。
    • 選択肢4（CloudTrail ベース）
      • CodePipeline のステージイベント検出に CloudTrail を使うのは不自然であり、EventBridge に比べて遅延や設定の複雑さが増す。

学習ポイント

  • GitHub 連携 CI/CD では、
    • CodePipeline + CodeBuild + CodeDeploy / S3
    • EventBridge でパイプラインステートチェンジを検出 → SNS 通知
      という流れを基本パターンとして覚える。
  • セキュリティレビューや PII チェックが要件に含まれる場合、静的コード解析ステージ を追加し、テストや手動承認ステージと同列に扱う構成をイメージする。
  • 通知は、まず SNS を使ってメール / ChatOps / 追加の Lambda 連携などに汎用的に拡張できる設計を優先する。
