問題65 不正解

シナリオ（日本語要約）

  • ある企業は、オンプレミスデータセンターのストレージを AWS VPC と Direct Connect で接続している。
    DevOps チームは Media Asset Management (MAM) システムを運用しており、50 TB を超えるデジタル動画・メディアファイルをオンプレミスのテープライブラリに保存している。
  • メディア資産が膨大なため、将来的には 顔認識を使った検索 を実現する自動カタログシステムを導入したい。
    カタログには、シーン内に登場する人物の名前と静止画像を含むメタデータを保存する。
    最終的には、動画コンテンツを AWS 上の MAM 動画コンテンツに移行したい。
  • 要件：
    • 継続的な運用オーバーヘッドを最小化 すること。
    • 既存システムへの 影響（中断）を最小限 に抑えること。
    • 上記を満たすソリューションを選ぶ。

質問

  • どの選択肢が、最小限の運用オーバーヘッドとシステム影響で要件を満たすか。

選択肢（日本語訳）

  • 選択肢1（正解）

    • オンプレミスファイルシステムを AWS Storage Gateway（ファイルゲートウェイ） と接続する。
    • MAM ソリューションを使って、現在のデータストアからメディアファイルを抽出し、ファイルゲートウェイへ送信する。
    • Amazon Rekognition の顔コレクションを作成し、処理済みのメディアファイルから 顔のカタログ を構築する。
    • AWS Lambda 関数 を起動し、Storage Gateway バックエンドの S3 バケットにあるメディアファイルに対して Rekognition JavaScript SDK を呼び出して顔を検出させる。
    • Lambda 関数で取得したメタデータを MAM ソリューションのカタログに格納する。

  • 選択肢2

    • Amazon Kinesis Video Streams を使用してビデオ取り込みストリームを作成し、Rekognition で顔コレクションを構築する。
    • MAM のメディアファイルを Kinesis Video Streams へストリーミングし、Rekognition で処理する。
    • ストリームコンシューマーを構成してメタデータを取得し、MAM に保存する。

  • 選択肢3（自分が選んだ不正解）

    • オンプレミスデータセンターに テープゲートウェイアプライアンス を起動し、AWS Storage Gateway へ接続する。
    • MAM ソリューションからメディアファイルを AWS Cloud 上のテープゲートウェイに送信し、そこから Rekognition 用の顔コレクションを構築する。
    • Lambda 関数で、テープゲートウェイからリアルタイムに Rekognition を呼び出して処理し、メタデータを MAM に保存する。

  • 選択肢4

    • すべてのメディアファイルをオンプレミスライブラリから大型 EC2 インスタンス上の EBS ボリューム へ移動し、オープンソースの顔認識ツールを起動してメタデータを取得する。
    • その後、メディアファイルを S3 バケットへコピーする。

正解の選択肢と注釈（選択肢1）

  • キーワード

    • AWS Storage Gateway File Gateway
    • オンプレミスファイルサーバー ↔ S3 バックアップ
    • Lambda + Rekognition による顔検出
    • 最小限のオペレーション + 既存ワークフロー継続

  • この解決策が要件を満たす理由

    • File Gateway はオンプレミスにキャッシュを持ちつつ、バックエンドを S3 にマッピングするため、MAM システムからは従来どおり ファイル共有としてアクセス できる。
    • 既存のワークフローを大きく変えずに、メディアファイルを S3 に徐々に取り込みつつ、Lambda + Rekognition で顔検出・メタデータ生成を自動化できる。
    • テープゲートウェイや大規模な EC2/EBS を新たに運用する必要がなく、継続的な運用オーバーヘッドが小さい。

  • 他の選択肢が不適切な理由（要点）

    • 選択肢2（Kinesis Video Streams）
      • ビデオストリームのリアルタイム処理向けであり、既存の 50 TB 以上のアーカイブメディアを処理するには 取り込みパイプラインの設計・運用が複雑。
    • 選択肢3（テープゲートウェイ、自分の誤答）
      • テープゲートウェイはバックアップ / アーカイブ用であり、ランダムアクセスや頻繁な読み取り を前提とした顔認識処理には不向き。
      • テープメディアからのリアルタイム処理はレイテンシが高く、アプリケーション統合も難しい。
    • 選択肢4（EC2 + オープンソース顔認識）
      • 独自の顔認識ソリューションを運用・スケールさせる必要があり、管理負荷・保守コストが大きい。
      • Rekognition のマネージドなスケーラビリティと精度を活用していない。

不正解と表示されている場合の追加注釈

  • 自分が選んだ不正解の選択肢

    • 選択肢3：テープゲートウェイを使ってリアルタイム処理 を選択した。

  • この選択肢が要件を満たさない理由

    • テープゲートウェイはアーカイブ用途であり、頻繁な読み書きやランダムアクセスを伴うワークロードには適さない。
    • 顔認識のような処理には File Gateway + S3 の方が、低レイテンシでオブジェクトベースの処理が可能 で運用オーバーヘッドも小さい。

  • 学習ポイント

    • Storage Gateway の 3 種類のモードの用途を整理する：
      • File Gateway：オンプレミスファイルサーバー ↔ S3 オブジェクトストレージ連携。
      • Volume Gateway：ブロックストレージ（iSCSI）バックアップ。
      • Tape Gateway：仮想テープライブラリによる長期アーカイブ。
    • アーカイブからの分析・検索など 頻繁な参照が想定される場合 は、Tape Gateway ではなく File Gateway / S3 への移行を優先する。
