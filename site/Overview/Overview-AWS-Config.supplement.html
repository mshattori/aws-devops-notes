🔍 1. 問題点・誤り・補足すべき箇所

  • Config ルールの評価タイミングと修復アクションの流れが簡略で、Periodic ルールやスコープ指定、ドリフト検出の用途に触れていない。
  • Aggregator の前提となる組織委任やクロスアカウント権限、配信チャネル（Delivery Channel/S3/CloudWatch Logs）への言及がなく、運用設計の視点が不足。
  • CloudTrail 監査や Config Recorder の有効化要件を明示せず、DOP 試験で問われる必須前提が抜けている。

🛠 2. 改訂版（修正・補足反映後の完成文）

  • 前提: 各リージョンで Config Recorder を有効化し、Delivery Channel で S3/CloudWatch Logs 出力を設定。CloudTrail 併用で変更履歴を完全取得。
  • Aggregator: Org 管理アカウントまたは委任アカウントで複数アカウント/リージョンのコンプライアンスを集約。クロスアカウント Role 付与が必要。
  • Config Rules: マネージド/カスタム（Lambda/Guard）を作成し、変更時 or Periodic (1–24h)。スコープ指定でリソースタイプ/タグ/ID を絞り込む。
  • 修復アクション: Noncompliant イベントで SSM Automation ランブックを呼び出し自動是正。必要に応じて Systems Manager 以外の通知やワークフローは EventBridge で分岐。
  • 代表ユースケース: CloudTrail 有効確認や暗号化設定の強制、ドリフト検出。評価結果は Compliant/Noncompliant としてダッシュボード/Exporter へ出力。
