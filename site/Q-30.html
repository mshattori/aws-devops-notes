Q-30

問題30 不正解

シナリオ（日本語要約）

  • ある分析用 Web ポータルは Amazon EMR クラスター と、それを取り巻く Auto Scaling グループ上の EC2 インスタンス を利用してデータ処理を行っている。
  • 3 か月後、AWS Trusted Advisor から「縮小（スケールダウン）が設定されていないため EMR インスタンスが遊休状態で高コストになっている」と推奨が出た。
  • 今後は Trusted Advisor からの推奨事項を 頻繁かつタイムリーに通知 し、同じ問題が再発しないようにしたい。

質問

  • Trusted Advisor の推奨をタイムリーに把握するために有効なソリューションとして、最も適切な組み合わせ（Select THREE） はどれか。

選択肢（日本語訳）

  • 選択肢1

    • Amazon EventBridge を作成し、Trusted Advisor チェックの変更をモニタリングする。
    • Amazon SES をトリガーとして設定し、メール通知を送信する。

  • 選択肢2（自分が選択し、正解でもある）

    • Trusted Advisor チェックステータスの変更をモニタリングする EventBridge ルール を作成する。
    • ルールのターゲットとして Amazon SNS を設定し、通知メールを送信する。

  • 選択肢3

    • Trusted Advisor の組み込み Amazon EventBridge 通知機能 を有効化し、チェック結果の変更をスキャンさせる。
    • アカウントオーナーに自動で通知を送信する。

  • 選択肢4（自分が選択し、正解でもある）

    • Trusted Advisor API を毎日実行する Lambda 関数 を作成し、チェックを更新する。
    • 結果に基づいてメッセージを SNS トピック へ公開し、サブスクライバーへ通知する。

  • 選択肢5（正解だが自分は選択しなかった）

    • Trusted Advisor の変更を毎日取得する Lambda 関数 を作成し、結果を CloudWatch Logs に送信する。
    • ログから CloudWatch Logs メトリクス を作成し、そのメトリクスに対する CloudWatch アラーム を構成して通知を送信する。

  • 選択肢6（自分が選んだ不正解）

    • Trusted Advisor の組み込み メール通知機能 を有効化し、通知メール（節約見積もりのサマリとチェック結果を含む）を自動受信する。

正解の選択肢と注釈（選択肢2・4・5）

  • キーワード

    • Trusted Advisor API / EventBridge / SNS
    • Lambda による定期実行
    • CloudWatch Logs メトリクス + アラーム

  • この組み合わせが要件を満たす理由

    • 選択肢2
      • Trusted Advisor イベントを EventBridge で直接トラッキングし、SNS メール によって即時通知できるため、もっともシンプルなイベント駆動型構成。
    • 選択肢4
      • Trusted Advisor API を Lambda で定期実行 することで、イベント型だけでなくスケジュール型のチェックも行える。
      • 結果を SNS に公開することで、複数チーム・複数チャネルへの通知配信が容易。
    • 選択肢5
      • Trusted Advisor 結果を CloudWatch Logs + メトリクス + アラーム に連携させることで、ダッシュボード表示やしきい値ベースのアラートを柔軟に定義できる。
      • 通知方法をメール以外（例えば PagerDuty 連携など）へ拡張しやすい。

  • その他の選択肢が不適切な理由（要点）

    • 選択肢1（EventBridge + SES）
      • SES で直接メール送信を行うと、SNS ベースの標準的な通知パターン から外れるうえ、サブスクライバーの増減管理が煩雑になる。
      • 問題文は SES を要求しておらず、よりシンプルな SNS を使うのがベストプラクティス。
    • 選択肢3（EventBridge 組み込み通知）
      • 「アカウントオーナーのみ」への通知に限定されやすく、運用チームへ柔軟に通知を配信できない。
      • 他の監視基盤（CloudWatch Logs や既存 SNS トピック）との統合がしづらい。

不正解だった自分の回答に対する注釈（選択肢6）

  • どの選択肢を誤って選んだか

    • 選択肢6：Trusted Advisor の組み込みメール通知機能 を選択した。

  • この選択肢が要件を満たさない理由

    • 組み込みメール通知は、主にコスト節約サマリや一部の推奨事項をまとめて通知するものであり、特定チェックの変化を頻繁かつタイムリーに検出 する用途には向かない。
    • 宛先が固定されやすく、複数のオペレーションチーム / チャネル へ柔軟に通知を配送する要件を満たしにくい。

  • 学習ポイント

    • Trusted Advisor を自動監視したい場合、まず EventBridge + SNS か Lambda + API + SNS/CloudWatch のパターンを思い出す。
    • 「Select THREE」問題では、組み込みの単純なメール機能だけでは運用要件（頻度・配布先・連携性）を満たさない ケースが多い。
    • CloudWatch Logs / メトリクス / アラームを絡めた構成は、長期的な可視性とアラート制御の柔軟性が高い。
